CRITICAL 2
Synchronized Thread.sleep() rate limiter blocks ALL servlet threads
src/main/java/com/animeSite/httpclient/JikanApiClient.java:62-71
Severity CRITICAL
Impact 5 concurrent users = all threads waiting on monitor; home page waterfall = 5×1s sleep sequential
Expected gain Thread contention eliminated; combined with @Cacheable, rateLimit() is rarely reached

synchronized void rateLimit() calls Thread.sleep() inside a servlet thread. Five API calls × 1s each = 5s before the first response byte. Under concurrent load all request threads queue behind the intrinsic lock.

// BETTER: Guava RateLimiter (non-blocking)
// build.gradle/pom.xml: com.google.guava:guava:33.x
import com.google.common.util.concurrent.RateLimiter;

private final RateLimiter rateLimiter = RateLimiter.create(1.0);

private void rateLimit() {
    rateLimiter.acquire(); // non-blocking — no Thread.sleep
}

// Even better: after @Cacheable, this is only called on cache miss — rare in production.
CRITICAL 3
saveAnime() fires 2N DB queries per list call (N+1 write pattern)
src/main/java/com/animeSite/service/impl/AnimeServiceImpl.java:126-148
Severity CRITICAL
Impact getTrending(25 items) = 25 SELECT + 25 INSERT overhead before returning to user
Expected gain -800ms DB time per list call; 1 batch query vs 50 individual queries

Every toAnimePage() call loops over Jikan results and calls saveAnime() for each — which does findByMalId() + save(). A read endpoint triggers 50+ writes. Fix: batch with findAllByMalIdIn() + saveAll().

// AnimeServiceImpl.java — batch upsert
private List<Anime> upsertBatch(List<JikanAnimeData> dataList) {
    Set<Integer> malIds = dataList.stream().map(JikanAnimeData::getMalId).collect(Collectors.toSet());
    Map<Integer, Anime> existing = animeRepository.findAllByMalIdIn(malIds)
        .stream().collect(Collectors.toMap(Anime::getMalId, a -> a));
    List<Anime> toSave = dataList.stream().map(d -> {
        Anime a = existing.getOrDefault(d.getMalId(), new Anime());
        a.setMalId(d.getMalId());
        // ... map all fields ...
        return a;
    }).toList();
    return animeRepository.saveAll(toSave); // single batch INSERT/UPDATE
}
CRITICAL 4
Image proxy has no cache — every image fetch hits MAL CDN synchronously through Spring
frontend/src/api/imageProxy.js  |  controller/ImageProxyController.java  |  frontend/.env (VITE_API_URL=render.com)
Severity CRITICAL
Impact Home page loads 60+ images through /api/image-proxy. Each is a live HTTP call to MAL. No Spring cache, no CDN.
Expected gain -2s LCP after CDN; -80% bandwidth after WebP + resize

The frontend routes every image through proxyImage(url) which returns ${API_BASE}/api/image-proxy?url=.... With VITE_API_URL=https://anime-watch-6h6z.onrender.com, every image first hits the Render server, which makes a synchronous HTTP GET to MAL CDN. No connection pooling, no ETag, no browser cache key. Also: import.meta.env.VITE_API_URL is set to Render.com but the Vite proxy proxies /api to localhost:8080. In dev mode these requests hit Render instead of the proxy — explaining the NS_BINDING_ABORTED errors when the Render server takes too long.

C. Major Refactors (1-3 days)

MAJOR 1
Replace per-request Jikan calls with scheduled import job

Every user request currently proxies through Jikan API (rate-limited, slow, unreliable). Instead: run a @Scheduled job every 30 minutes that imports trending/seasonal/search results into PostgreSQL. User-facing endpoints only query the DB — sub-5ms instead of 1-5s.

// JikanSyncJob.java — scheduled batch import
@Component
public class JikanSyncJob {
    @Scheduled(cron = "0 */30 * * * *")
    @Transactional
    public void syncTrending() {
        JikanListResponse response = jikanApiClient.fetchTopAnime(0);
        upsertBatch(response.getData());
    }

    @Scheduled(cron = "0 */30 * * * *")
    @Transactional
    public void syncSeasonal() {
        JikanListResponse response = jikanApiClient.fetchSeasonalAnime(0);
        upsertBatch(response.getData());
    }
}

// AnimeServiceImpl — now DB-only, 5ms instead of 1-5s
@Override
@Cacheable("trending")
public AnimePage getTrending(int page) {
    Page<Anime> dbPage = animeRepository.findAllByOrderByRatingDesc(PageRequest.of(page, 25));
    return toAnimePageFromDb(dbPage, page);
}
Eliminates Jikan rate limit contention entirely. User endpoints become 5-10ms DB reads.
MAJOR 2
Add Redis distributed cache (replace Ehcache in-process)

Current Ehcache is per-instance — restarts clear cache, multi-pod deployments thrash Jikan. Redis provides shared cache across instances with sub-ms read latency.

// pom.xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>

// application.yml — swap cache provider, annotations stay same
spring.cache.type: redis
spring.data.redis.host: ${REDIS_HOST:localhost}
spring.data.redis.port: ${REDIS_PORT:6379}
Cache survives restart, shared across instances, sub-ms read latency
MAJOR 3
Replace data fetching with TanStack Query

Every page has the same pattern: useState + useEffect + manual loading/error. React Query eliminates all boilerplate with: stale-while-revalidate caching, automatic deduplication, background refetch, query invalidation on mutations, and pagination support. Also eliminates the manual withCache() wrapper.

// Home.jsx — 5 API calls in 15 lines
const queries = useQueries({
  queries: [
    { queryKey: ['trending'], queryFn: () => getTrending(0,25), staleTime: 300_000 },
    { queryKey: ['seasonal'], queryFn: () => getSeasonal(0,20), staleTime: 300_000 },
    { queryKey: ['airing'], queryFn: () => filterAnime({status:'airing'}), staleTime: 300_000 },
    { queryKey: ['complete'], queryFn: () => filterAnime({status:'complete'}), staleTime: 300_000 },
    { queryKey: ['upcoming'], queryFn: () => filterAnime({status:'upcoming'}), staleTime: 300_000 },
  ]
})

const [trending, seasonal, airing, complete, upcoming] = queries.map(q => q.data?.data ?? []);
// Each query has isLoading, error, data — render per-section independently
-70% fetch boilerplate, auto-dedup, instant back-navigation, cache invalidation on mutations
MAJOR 4
Virtualize episode list in WatchPage

Long-running anime render 500+ episode button DOM nodes. Use @tanstack/react-virtual to render only visible ones (18 instead of 500+).

import { useVirtualizer } from '@tanstack/react-virtual'

const rowVirtualizer = useVirtualizer({
  count: episodes.length,
  getScrollElement: () => parentRef.current,

MAJOR 5
CDN for images (Cloudflare + imgproxy)

Every anime image currently routes through Spring Boot's /api/image-proxy. At 60+ images per home page this creates 60 backend connections. Use a CDN image proxy (Cloudflare Images, Imgix, imgproxy) for on-the-fly WebP conversion + resizing + caching at edge. Frontend builds CDN URLs directly — zero Spring involvement.

-2s LCP, -80% bandwidth (WebP + 360px resize), eliminates 60+ Spring threads per home load
